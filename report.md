# Тестирование и мониторинг ML-сервиса

## 1. Доработка надёжности и отказоустойчивости сервиса

Сервис был доработан так, чтобы в проблемных ситуациях возвращать корректные HTTP-ответы с понятными сообщениями, а не завершаться с `500 Internal Server Error`.

### Обработанные проблемные ситуации

1. **Невалидные типы входных данных в `/predict`**
   - Пример: `"age": "abc"`
   - Ошибка обрабатывается через валидацию FastAPI/Pydantic, сервис возвращает `422 Unprocessable Entity`.

2. **Пустой JSON в `/predict`**
   - Пример: `{}`
   - Добавлена проверка, из-за которой сервис возвращает `422`, а не падает внутри sklearn pipeline.

3. **Отсутствие обязательных признаков для текущей модели**
   - Если для активной модели не переданы нужные фичи или в них есть `null`, сервис возвращает `422` и список проблемных полей.

4. **Модель не загружена**
   - Если модель отсутствует, `/predict` возвращает `503 Service Unavailable`.

5. **Пустой `run_id` в `/updateModel`**
   - Возвращается `422` с понятным сообщением об ошибке.

6. **Невалидный или несуществующий `run_id` в `/updateModel`**
   - Возвращается `404` с описанием ошибки загрузки модели вместо падения сервиса.

7. **Ошибки предобработки и инференса**
   - Исключения во время подготовки данных и работы модели перехватываются и преобразуются в корректные HTTP-ответы.

## 2. Написанные тесты

Для сервиса были написаны тесты с использованием `pytest` и `fastapi.testclient`.

### API-тесты
- `test_health_ok`
  - проверяет, что `/health` возвращает `200 OK`
- `test_predict_ok`
  - проверяет успешный инференс для корректного запроса
- `test_predict_invalid_type`
  - проверяет валидацию неверного типа входных данных
- `test_predict_empty_json`
  - проверяет обработку пустого запроса в `/predict`
- `test_update_model_empty_run_id`
  - проверяет обработку пустого `run_id` в `/updateModel`

### Тесты предобработки данных
- `test_to_dataframe_all_columns`
  - проверяет корректное преобразование полного входа в `DataFrame`
- `test_to_dataframe_only_needed_columns`
  - проверяет корректный выбор только нужных колонок

### Что покрывают тесты
Тесты покрывают:
- функции предобработки данных
- логику хэндлеров и валидацию запросов
- успешную загрузку состояния модели и инференс через API

## 3. Логирование метрик

Для мониторинга был выбран **Prometheus**.  
В сервис добавлен endpoint `/metrics`, который отдает метрики в формате Prometheus.

### Технические метрики сервиса
- `ml_service_requests_total`
  - общее число HTTP-запросов с разбивкой по endpoint, методу и status code
- `ml_service_request_latency_seconds`
  - гистограмма времени ответа сервиса
- `ml_service_cpu_percent`
  - текущая загрузка CPU
- `ml_service_memory_percent`
  - текущее потребление памяти

### Метрики, связанные с данными
- `ml_service_preprocessing_seconds`
  - время предобработки входных данных
- `ml_service_predict_requests_total`
  - число predict-запросов с разбивкой по результату обработки (`success`, `null_values`, `missing_columns` и др.)

### Метрики, связанные с моделью
- `ml_service_inference_seconds`
  - время инференса модели
- `ml_service_predictions_total`
  - количество предсказаний по классам
- `ml_service_prediction_probability`
  - распределение вероятностей, возвращаемых моделью

### Метрики, связанные с моделью и её обновлением
- `ml_service_model_updates_total`
  - число попыток обновления модели с разбивкой по статусу
- `ml_service_model_info`
  - информация о текущей модели (`run_id`, тип модели)
- `ml_service_model_feature_count`
  - число признаков, требуемых активной моделью

### Перцентили
Для временных метрик использовались гистограммы, которые позволяют строить в Grafana панели по перцентилям, в частности:
- p95 для времени ответа сервиса
- p95 для времени предобработки
- p95 для времени инференса

## 5. Алерты

В Grafana были настроены следующие alert rules:

1. **Predict errors detected**
   - срабатывает при появлении неуспешных predict-запросов

2. **High response latency**
   - срабатывает при превышении порога по времени ответа сервиса

3. **High inference latency**
   - срабатывает при превышении порога по времени инференса модели

4. **High CPU**
   - срабатывает при превышении порога по загрузке CPU

5. **5xx errors detected**
   - срабатывает при появлении серверных ошибок 5xx

Эти алерты покрывают как техническое состояние сервиса, так и ошибки, связанные с работой ML-пайплайна.

## 6. Мониторинг дрифта с использованием Evidently

В сервис был добавлен мониторинг дрифта входных данных и предсказаний модели при помощи Evidently.

### Реализованный подход
- при старте сервиса создается reference dataset
- успешные запросы `/predict` сохраняются в памяти как current data
- вместе с признаками сохраняются:
  - предсказанный класс
  - вероятность
- в фоне запускается корутина, которая периодически:
  - проверяет, накопилось ли достаточное число записей
  - строит drift report
  - отправляет его в Evidently workspace

### Используемый workspace
- URL: `http://158.160.2.37:8000/`
- использован созданный в интерфейсе Evidently проект

### Проверка работоспособности
Успешная отправка отчета подтверждена логом сервиса:

`Evidently report sent successfully. reference=30, current=12`

После этого отчет появился в интерфейсе Evidently в разделе `Reports`.
